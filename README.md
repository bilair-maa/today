# Today: Take-Home Submission

## Overall Approach

Before opening any tools, I read the spec with attention to both the technical requirements and the overall product concept. The core constraint, that the app only cares about today, pointed toward a specific kind of user: someone dealing with anxiety, ADHD, or the kind of overwhelm that comes from a task list that grows increasingly faster. That read on the target user informed every product and technical decision that followed.

With this in mind, I thought about what the UI should feel like before writing any code. For a user who is already prone to overwhelm, visual noise is a real problem, and I wanted the interface to reflect that. A good reference point is the difference between Google and Yahoo, where Yahoo had everything on the page competing for attention and Google was just a blank input. I used Claude Design to put together initial mockups once I had that direction clear, providing specific constraints around layout and tone so the output matched the vision I had already defined. With more time I would have been more personally involved in that process, because I think visual design benefits from a more iterative and hands-on approach than prompting alone allows.

For implementation I used Claude Code as my primary tool, working through the codebase file by file. One of the most important things I was conscious of throughout was staying token-aware and keeping prompts scoped to what I actually needed, because broad or vague context produces output that drifts from the architecture and creates more work than it saves. I reviewed every output before accepting it and pushed back when something was added that I had not asked for or that introduced unnecessary complexity. For syntax-level edits and smaller targeted changes when I was working directly in the code, I used GitHub Copilot. Security was something I actively reviewed for at every step, making sure private user data like task titles never appeared in logs, that input validation lived in the ViewModel where it could not be bypassed through a secondary entry point, that nothing read from UserDefaults was force unwrapped.

In terms of build order, I set up the architecture and project structure first, then built out the must-have features before touching anything related to polish or optional functionality.

## Key Decisions and Tradeoffs

The most consistent tension throughout the build was between giving users flexibility and keeping the experience focused enough to serve its purpose, and every feature decision came back to that question.

For architecture I used MVVM with a strict layer rule where views talk to the ViewModel, the ViewModel talks to the TaskStore, and nothing skips a layer. The TaskStore is the only file that touches UserDefaults, so persistence is fully contained in one place. The TaskViewModel owns all validation and orchestration logic, and the day reset is triggered explicitly through onAppLaunch() rather than inside the store's init(), because calling it from init() would mean it fires on every instantiation including in tests, making it impossible to set up a controlled state before triggering it. Input validation living in the ViewModel also means it applies automatically if a second entry point like a widget or Siri shortcut is added later, rather than having to be duplicated across multiple call sites.

On the UI side, task titles are capped at 280 characters and descriptions at 500, with live character counts that appear as users approach the limit and inline error states that surface immediately rather than waiting for a form submission. The day start hour is constrained to a sensible range rather than allowing arbitrary input, and expiration times that fall outside the user's defined day window are caught and corrected with clear feedback rather than silently accepted.

For persistence I chose UserDefaults with Codable because SwiftData requires iOS 17 and the minimum target is iOS 16, and a flat task list with no relational complexity does not need a full Core Data stack. Category IDs are stable from the moment they are first written to UserDefaults on launch rather than being regenerated from static instances, which prevents saved tasks from losing their category references across sessions. 

A particular custom feature I implemented was a configurable day start hour feature that came from a recognition that midnight is not a universal boundary. Screen Time is a good example of this: if you are a night owl and set a daily limit on an app, that limit count starts at midnight while you are still awake and still in what feels like the same day, so the limit is reached before you have even gone to sleep. For an app built around the concept of today, having the same assumption baked in would undermine the whole point for a meaningful portion of users. The today-only constraint is fully preserved, tasks still belong to a single 24-hour window and reset at the end of it, but that window is anchored to when the user's day actually begins rather than an arbitrary clock time. I capped the setting at 10 AM and added a confirmation step before the change so as to avoid the risk of the feature being used in ways that work against the users. The goal was always to give people a reasonable range that accommodates real schedules rather than an escape hatch. The confirmation step is there because changing when your day resets is a meaningful decision, and a small moment of friction ensures it is an intentional one rather than something adjusted casually and repeatedly.

On the product side, I implemented custom categories with the ability to add, rename, and delete them, but capped the total number so the list stays manageable rather than becoming another source of clutter. Tasks support priority levels, optional expiration times, and reminder offsets so users can set a notification to fire at a chosen interval before a tasks assigned expiration. Notifications are also tied to priority level, with high priority tasks receiving multiple reminders across the day and medium priority tasks receiving one, which keeps the system useful without overwhelming the user with alerts. The filtering system lets users filter the task list by one or more categories simultaneously, and tasks can be manually reordered via a drag handle and edited after creation, so the list stays accurate as the day changes. Tasks also have haptic feedback tied to key interactions like checking off a task.

I had also intentionally chose to leave out expired task history even though it is listed as an optional enhancement, because surfacing incomplete expired tasks recreates the same backlog pressure the app is built around avoiding. I also considered an all-time completion counter as a motivational feature but decided against it because cumulative tracking works against the philosophy of treating each day as a clean slate. Both decisions were product calls.

The visual system is built on a set of semantic color tokens across four theme variants, warm light, warm dark, cool light, and cool dark, which keeps theming consistent and makes it a single-point change rather than hunting values across files. The four variants also give users a degree of personalization without the decision fatigue that comes with fully open customization. A great larger scale example is the Android vs. Apple comparison: Android offers expansive control, while Apple has a more constrained palette, which to some is a part of its appeal precisely because it removes the burden of choice. The final state when all tasks are completed was similarly intentional: the list clears and the screen opens up, meant to feel like a clean slate. The option to add tasks or view completed ones is still there, but the language is framed to feel encouraging rather than prescriptive, invoking a sense of accomplishment rather than implying there is more to be done.

## What I Would Improve With More Time

With more time, I would add comprehensive unit tests covering the ViewModel validation logic, the day reset behavior, input constraints, and the configurable day boundary, and I would do more thorough QA across edge cases in notification behavior, widget state, and category management. I would also be more personally involved in the visual design process. I would also love to implement some other feature ideas I had, that I did not have the chance to incorporate, including some of the optional feature suggestions.

On the AI workflow side, I would have been more intentional about session management if I had more time. One of the things I feel strongly about is that AI is a tool, and like any tool the quality of the output depends entirely on how deliberately you use it. Knowing how to prompt well is not just about getting cleaner code, it directly affects token usage, context quality, and how much rework ends up being necessary. I was deliberate about this throughout the build, but with more time I would keep sessions even shorter and more scoped, start fresh when switching between features to prevent context drift, and be more precise about what I asked AI to generate versus what I reasoned through myself. I think the distinction between using AI intentionally and vibe coding is an important one, and it is something I am actively conscious of and would continue to prioritize.

## What I Got Stuck On

When I was working on the data models, particularly the category UUID stability. The Default categories were generating a new identifier on every app launch, meaning saved tasks would silently lose their category references across sessions. I caught this through code review rather than the AI flagging it, and the fix was treating default categories as seed data written once on first launch rather than regenerated from static instances. Another example was a nil check in the day reset logic that was functionally harmless but logically incorrect. The AI had accepted it without comment and I caught it reading through the code myself. Another moment was mid-build, a question about whether the ViewModel was genuinely adding value or just forwarding calls to the store.  It came up because the AI kept placing logic in the wrong layer. I stopped to verify, traced through what each layer was responsible for, and confirmed the separation was correct.

Beyond the technical issues, some of the harder moments were product decisions I was going back and forth with myself on. The expired task history is the good example: it is a natural feature for a task app, but I kept coming back to whether surfacing incomplete expired tasks would recreate exactly the kind of backlog pressure the app is built around avoiding, and I eventually decided it would. The configurable day start hour was similar: I went back and forth on whether it violated the today-only constraint before landing on the reasoning that the constraint is about a 24-hour window, not midnight specifically, and that anchoring it to when a user's day actually begins serves the product better.

A recurring theme on the AI side was pushing back and cleaning up after it. It reached for newer APIs that did not meet the iOS 16 requirement, and at points suggested collapsing architectural layers or adding abstractions I had not asked for. One example is when implementing custom categories, the AI insisted the feature was too complex, suggested avoiding it entirely, and at one point told me the approach I had in mind was not possible even though it was. Holding that line meant being specific not just about what I wanted but about why, and understanding the code well enough to know the complexity argument was not valid and the implementation was straightforward. Dead code was another consistent issue: when I removed a feature or changed direction, the AI was not good at cleaning up after itself, and I made a habit of going back through the codebase to remove anything no longer referenced or relevant. Beyond the code itself, I ran through the app in the simulator consistently, not just spot checking, but walking through real usage flows to catch anything that worked in isolation but broke in context, whether that was UI behavior, state transitions, or the day reset logic behaving correctly across edge cases.

## On the Use of AI
One thing I feel strongly about is that AI is a tool, not a replacement, and like any tool the quality of the output depends entirely on how deliberately you use it. Different models have different strengths, and part of using AI well is knowing which one to reach for and when. For this project I built a small stack around that idea: Claude Code for implementation sessions, GitHub Copilot for targeted syntax-level edits directly in the editor, Claude web for reasoning through decisions, and Claude Design for the initial UI mockups. Each served a specific purpose rather than one tool handling everything.

The files provided in the UseofAI folder reflect that mix. Some sessions are tight and scoped, which is closer to how I think good AI prompting should work. Others are more conversational or voice-dictated, which is an honest reflection of building under time pressure alongside other commitments. In a normal working context I would be more deliberate about session length, prompt specificity, token efficiency, and knowing earlier when starting a fresh session would be cleaner than correcting course mid-conversation. The Claude.md context file included with the submission is probably the clearest example of that intentionality, I wrote it before any implementation sessions began to define the security constraints, architectural boundaries, and code quality standards I wanted consistent across every session, so those decisions were made once rather than relitigated in every conversation.
