Good set of ideas. Here's my take on each, then you can tell me what to build:

**Notifications strategy — priority drives frequency.** This is the cleanest approach and avoids a separate "reminder" picker that most users won't configure per-task. My recommendation:
- **High** = 3 nudges/day (morning, midday, evening)
- **Medium** = 1 nudge/day (midday check-in)
- **Low** = 0 nudges (just the general end-of-day summary)
- **All tasks** get a single end-of-day "wrap-up" notification (~2 hours before day-end) saying something like "3 of 5 done — you've got this"
- If a task has an **expiration**, it also gets a 30-min-before reminder regardless of priority

This means no extra UI in AddTaskView for reminders — priority *is* the reminder level, which is elegant and easy to explain in onboarding.

**Notification permissions** — a toggle in Settings ("Notifications: On/Off") that triggers the system permission prompt on first enable. We'd use `UNUserNotificationCenter` to schedule local notifications. No server needed.

**Task sorting** — the standard in task apps (Todoist, Things, TickTick) is **priority first, then expiration** (soonest deadline first within the same priority tier). Rationale: a high-priority task with no deadline still outranks a low-priority task expiring soon. This matches user intent — they chose "high" for a reason.

**Onboarding tutorial** — 3-4 swipeable full-screen pages shown on first launch. Would cover: (1) add your tasks for today, (2) priority levels control reminders, (3) tasks reset each day, (4) enable notifications. Minimal, matches the app's typographic style.
